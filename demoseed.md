# Demo seed (фаза 15)

Спецификация консольной команды для наполнения БД demo-данными (3 вселенные × ~40 заметок) — dev, скринкаст, ручная проверка графа связей, фильтрации по тегам и истории версий.

**Ограничения:** новые Doctrine-сущности **не нужны**; не ходить через API Platform / HTTP — прямая работа с EntityManager и существующими сервисами.

## Команда

| Параметр | Значение |
|----------|----------|
| Имя | `app:seed-demo-data` |
| Флаг | `--force` — удалить demo-пользователей по email и пересоздать данные |
| Без `--force` | если любой из demo-email уже есть — предупреждение и exit 1 (не трогать чужие данные) |

**Demo-пользователи** (общий пароль `demo1234`):

| Email | Вселенная | Роль |
|-------|-----------|------|
| `hogwarts@demo.local` | Гарри Поттер | `ROLE_USER` |
| `westeros@demo.local` | Игра престолов | `ROLE_USER` |
| `witcher@demo.local` | Ведьмак | `ROLE_USER` (+ опционально `ROLE_ADMIN` для проверки админки) |

Удаление при `--force`: `UserRepository::findOneBy(['email' => …])` → `$em->remove($user)` → `flush()`; каскад через `orphanRemoval` на `notes`/`folders`/`tags`, FK на `note_links`/`note_versions` — CASCADE.

**Типичный dev-сценарий:**

```bash
docker compose exec php php bin/console app:reset-schema
docker compose exec php php bin/console app:seed-demo-data
# опционально: app:create-admin для отдельного админа из .env
```

## Структура кода (backend)

```
backend/src/
  Command/
    SeedDemoDataCommand.php          # CLI: --force, вывод статистики
  DemoSeed/
    DemoUniverseDefinition.php       # DTO: email, roles, folders[], tags[], notes[]
    DemoNoteDefinition.php           # key, title, folderPath?, content, isFavorite?, tags[], versions[]?
    DemoVersionDefinition.php        # title, content, createdAtOffset (напр. '-7 days')
    DemoUniverseSeeder.php           # оркестратор одной вселенной
    Universe/
      PotterUniverse.php             # implements/returns DemoUniverseDefinition
      WesterosUniverse.php
      WitcherUniverse.php
```

- **Нет** отдельных Entity, **нет** Doctrine Fixtures bundle — только PHP-массивы/heredoc markdown в классах вселенных.
- Каждый `*Universe.php` — статический метод `definition(): DemoUniverseDefinition` (~800–1500 строк markdown суммарно на вселенную — нормально).

## Алгоритм `DemoUniverseSeeder`

1. Создать `User`, захешировать пароль (`UserPasswordHasherInterface`).
2. **Папки:** обойти дерево из definition (`['Хогвартс', 'Факультеты']` → path key `'Хогвартс/Факультеты'`); map `path → Folder`.
3. **Теги:** для каждого имени из `DemoUniverseDefinition::tags` создать `Tag` (user + name); map `tagName → Tag`. Словарь тегов **уникален для вселенной** — не переиспользовать между пользователями.
4. **Заметки (pass 1):** для каждой `DemoNoteDefinition`:
   - создать `Note` с `title`, `content` (пока с плейсхолдерами), `folder`, `isFavorite`;
   - привязать теги из `tags[]` через `$note->addTag()`;
   - map `noteKey → Note`.
5. **Wiki-ссылки (pass 2):** в content заменить плейсхолдеры на реальные UUID:
   - синтаксис в definition: `{{link:harry}}` → `[[uuid]]`, `{{link:harry|Мальчик-который-выжил}}` → `[[uuid|alias]]`;
   - regex replace по map `noteKey → (string) note->getId()`.
6. **Links:** для каждой заметки вызвать `NoteLinkSyncService::syncFromContent($note)` (не дублировать логику парсера).
7. **Версии (pass 3):** для заметок с непустым `versions[]`:
   - persist `NoteVersion` напрямую через EntityManager (не через `NoteVersionService::recordVersionOnUpdate` — это логика PUT);
   - `createdAt` = `now + createdAtOffset` (от `-14 days` до `-1 day`), сортировка от старых к новым;
   - текущие `title`/`content` заметки — **актуальное** состояние; `versions[]` — более ранние снимки (видно эволюцию текста при restore/preview).
8. Один `flush()` в конце вселенной (или batch по 20 заметок — по желанию при OOM).

Плейсхолдеры вместо UUID в definition — чтобы не генерировать контент вручную с заведомо несуществующими id и упростить читаемость фикстуры.

## Теги по вселенным

Теги объявляются в `DemoUniverseDefinition::tags` и назначаются заметкам через `DemoNoteDefinition::tags` (1–3 тега на заметку). Имена — **тематические для вселенной**, не общие вроде «важное» / «черновик».

| Вселенная | Словарь тегов (~8–12) | Примеры назначения |
|-----------|----------------------|--------------------|
| Potter | `персонаж`, `факультет`, `заклинание`, `артефакт`, `событие`, `хогвартс`, `тёмная-магия`, `квиддitch`, `учебник` | Гарри → `персонаж`, `гриффиндор`; Expelliarmus → `заклинание`; Битва при Хогвартсе → `событие`, `тёмная-магия` |
| Westeros | `дом`, `персонаж`, `локация`, `война`, `интрига`, `север`, `юг`, `драконы`, `ночной-дозор` | Старки → `дом`, `север`; Красная свадьба → `интрига`, `война`; Стена → `локация`, `ночной-дозор` |
| Witcher | `ведьмак`, `чудовище`, `локация`, `квест`, `алхимия`, `знак`, `контракт`, `монстрология` | Геральт → `ведьмак`; Стрыга → `чудовище`, `контракт`; Кошачий → `знак`, `алхимия` |

Ориентиры:

- у каждого пользователя **8–12 тегов** в словаре;
- **каждая заметка** получает хотя бы один тег;
- **2–4 заметки** с пересечением тегов (фильтр «И» в sidebar) — напр. `персонаж` + `север`, `квест` + `чудовище`;
- hub-заметки — тег `хогвартс` / `дом` / `ведьмак` (или аналог по вселенной).

## История версий

Версии нужны для демо UI «История версий» в тулбаре заметки. Чтобы быстро найти такие заметки — **все заметки с версиями помечать `isFavorite: true`**.

| Параметр | Значение |
|----------|----------|
| Заметок с версиями на вселенную | **3–5** (hub + 1–2 главных персонажа + 1 «живая» заметка с правками) |
| Снимков в `versions[]` | **2–4** на заметку |
| `isFavorite` | **обязательно `true`** у всех заметок с `versions[]` |
| Прочие избранные | ещё **2–4** заметки без версий (хабы / персонажи для вкладки «Избранное») |

Примеры заметок с версиями:

| Вселенная | Заметка | Идея эволюции |
|-----------|---------|---------------|
| Potter | «Краткий путеводитель» (hub) | черновик → дополнены факультеты → добавлены wiki-ссылки |
| Potter | «Гарри Поттер» | краткая карточка → биография → текущая с ссылками на события |
| Westeros | «Печать: зима близка» (hub) | список домов → добавлены войны → актуальный обзор интриг |
| Westeros | «Нед Старк» | первая версия до событий → после «Красной свадьбы» (контент меняется) |
| Witcher | «Записи бестиария» (hub) | оглавление → разделы по типам → полный бестиарий |
| Witcher | «Геральт из Ривии» | заметка контрактника → дополнены знаки и квесты |

В definition у такой заметки:

```php
new DemoNoteDefinition(
    key: 'harry',
    title: 'Гарри Поттер',
    isFavorite: true,
    tags: ['персонаж', 'гриффиндор'],
    content: '…актуальный текст…',
    versions: [
        new DemoVersionDefinition('Гарри', '…краткая карточка…', '-12 days'),
        new DemoVersionDefinition('Гарри Поттер', '…расширенная биография…', '-5 days'),
    ],
),
```

## Граф wiki-ссылок (идея связности)

Цель — локальный граф depth=2 выглядит насыщенно; не полный mesh.

- **1–2 hub-заметки** на вселенную — много исходящих ссылок (`[[uuid]]` без alias).
- **Персонажи** ссылаются друг на друга и на hub; часть ссылок с alias (`[[uuid|Профессор]]`).
- **События** — несколько ссылок на одних персонажей (проверка `aliases.length > 1` в 14.2).
- **3–5 заметок** только входящие ссылки (leaf) — для `incoming` badge.
- **Битых ссылок нет** — все `{{link:key}}` резолвятся в map.

Ориентир: ~60–90 исходящих `note_links` строк на пользователя (не 40, т.к. несколько ссылок на один target сливаются в одну строку с массивом aliases).

## Markdown в заметках

- H2/H3, списки, blockquote, **жирный**, `inline code`
- **Без смешения латиницы и кириллицы в одном слове** (напр. «Пuffenduy» — нельзя; либо «Пуффендуй», либо целиком латиницей «Hufflepuff»). Имена заклинаний целиком на латинице (`Expelliarmus`) — допустимы; ключи `{{link:key}}` — латиница; пользовательский текст и теги — кириллица.
- 1–2 заметки со structured списками (дома × девиз × регион; знаки × эффект) — **без markdown-таблиц** (редактор их не поддерживает)
- 1 заметка с fenced code block (например «рецепт зелья» / «клятва Night's Watch»)
- Длина тела: 200–800 символов в среднем, hub — до 1200
- В старых **версиях** hub/персонажей — более короткий текст без части wiki-ссылок (эволюция контента)

## Что сознательно не делаем

- Backfill / отдельная команда — только `app:seed-demo-data`.
- Фронтенд — без изменений.
- Версии через `NoteVersionService::recordVersionOnUpdate` — только прямой persist `NoteVersion` в seeder.
