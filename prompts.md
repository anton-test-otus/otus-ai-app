   - Созданы endpoints:

6. **Базовый CRUD для заметок с пагинацией:**

   - Настроена пагинация: 20 элементов по умолчанию, максимум 100
   - Endpoints для Note:
     - `GET /api/notes` - список с пагинацией
     - `GET /api/notes/{id}` - получение заметки
     - `POST /api/notes` - создание
     - `PUT /api/notes/{id}` - обновление
     - `DELETE /api/notes/{id}` - удаление
   - Созданы кастомные методы в NoteRepository для фильтрации

7. **Администрирование пользователей:**
   - Создан `AdminController` с endpoints:
     - `GET /api/admin/users` - список пользователей (пагинация)
     - `GET /api/admin/users/{id}` - информация о пользователе

## Промпт 114

> так. а почему бы не сделать при сохранении ТОЛЬКО [[, без экранирования?
> или в этом есть какие-то проблемы?

### Выполненные действия:

**Почему было `\[\[uuid\]\]`**
- При сохранении wiki-ссылка сериализовалась как mdast `text`-узел с `[[uuid|alias]]`
- `mdast-util-to-markdown` экранирует `[` в тексте (`state.safe`), т.к. `[` — синтаксис markdown-ссылок
- Отсюда `\[\[uuid|alias\]\]` в БД — не намеренный формат, а побочный эффект сериализации

**Проблем с `[[uuid|alias]]` для приложения нет**
- Это наш кастомный синтаксис; парсится remark-плагином
- Ограничения те же, что и раньше: `|` и `]]` в alias ломают формат
- В сторонних markdown-редакторах ссылка будет видна как текст — для нас это нормально

**Исправление сериализации**
- `toMarkdown` теперь пишет mdast-узел `wikiLink`, а не `text`
- `remarkWikiLinkStringifyPlugin` — handler в `toMarkdownExtensions`, выводит `[[uuid|alias]]` без экранирования
- Подключён в редакторе и preview

**Результат:** новые сохранения пишут в БД чистый `[[uuid|alias]]`; поддержка экранированного формата в preview/parser остаётся для старых заметок до их удаления.
